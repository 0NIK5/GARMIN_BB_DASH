 



Garmin Recovery Dashboard — Technical Specification

1. Product Vision

Легковесный дашборд для мониторинга физиологического состояния пользователя на основе метрики Body Battery из Garmin.

Цель:
	•	отслеживание Body Battery
	

SLA:
	•	задержка данных ≤ 60 минут

Ограничения:
	•	используется неофициальный API 
	•	single-user MVP



2. System Architecture

Компоненты
	1.	Data Worker
	2.	API Backend
	3.	Database
	4.	Frontend

Архитурная схема (логическая)

Garmin → Worker → Database → API → Frontend




3. Data Flow (End-to-End)
	1.	Worker запускается по cron (каждый час, HH:05)
	2.	Загружает данные:

[last_saved_timestamp - 1 hour, now]


	3.	Фильтрует новые записи
	4.	Делает UPSERT в БД
	5.	API читает данные из БД
	6.	Frontend опрашивает API каждые 5 минут



4. Tech Stack

Layer	Technology
Worker	Python 3.10+, garminconnect
Scheduler	cron / APScheduler
Backend	FastAPI + SQLAlchemy
Database	 SQLite (MVP)
Frontend	HTML + Tailwind + Chart.js
Infra	Docker Compose



5. Data Worker

5.1. Основная логика

start = last_timestamp - 1 hour
end = now

data = client.get_body_battery(start, end)




5.2. Session Management

Требования:
	•	логин выполняется только при первом запуске
	•	сессия сохраняется в session.json

Поведение:

Ситуация	Действие
session валидна	использовать
401 / 403	перелогин + перезапись
ошибка	retry




5.3. Retry Policy
	•	network error:
	•	3 попытки
	•	backoff: 5s → 15s → 30s
	•	401/403:
	•	refresh session
	•	retry 1 раз
	•	после неудачи:
	•	логирование
	•	graceful exit



5.4. Deduplication Strategy
	•	уникальность: measured_at
	•	стратегия:

ON CONFLICT (measured_at) DO NOTHING




5.5. Logging (Worker)

Обязательно логировать:
	•	старт/конец job
	•	количество записей
	•	ошибки Garmin API
	•	попытки retry



5.6. Failure Handling

Сценарий	Поведение
Garmin недоступен	retry → exit
session invalid	refresh
DB недоступна	retry → exit




6. Database Schema

Таблица: body_battery_logs

Поле	Тип	Ограничения
id	SERIAL	PK
measured_at	TIMESTAMP (UTC)	UNIQUE, INDEX
level	SMALLINT	5–100
fetched_at	TIMESTAMP (UTC)	NOT NULL

Индексы

CREATE UNIQUE INDEX idx_measured_at ON body_battery_logs(measured_at);
CREATE INDEX idx_measured_at_desc ON body_battery_logs(measured_at DESC);




Time Handling
	•	хранение: UTC
	•	API: ISO 8601 (Z)
	•	фронтенд: локальная конверсия



7. API Specification

Base URL

/api/v1




7.1. GET /battery/current

Response (200)

{
  "timestamp": "2026-04-08T10:05:00Z",
  "level": 52,
  "status": "draining",
  "minutes_since_update": 7,
  "is_stale": false
}

Логика:
	•	status:
	•	рассчитывается по последним 3 точкам
	•	is_stale:
	•	true если > 90 минут без обновления



7.2. GET /battery/history?hours=24

Response (200)

{
  "period_hours": 24,
  "data": [
    {"time": "2026-04-07T13:00:00Z", "level": 88}
  ]
}




7.3. Errors

Формат

{
  "error": "DATA_NOT_AVAILABLE",
  "message": "No data for the requested period"
}

Коды

Код	Значение
404	нет данных
500	ошибка сервера
503	источник недоступен




7.4. Performance
	•	target latency: < 200ms
	•	read-only API

13. Definition of Done

Фича считается завершённой, если:
	•	Worker пишет данные ≥ 24 часов без сбоев
	•	нет дубликатов
	•	API отвечает < 200ms
	•	фронтенд отображает данные корректно
	•	система поднимается через:

docker-compose up
